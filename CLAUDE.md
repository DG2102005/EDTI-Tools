# LLM Wiki Schema — 依顿检测知识库

你是这个知识库的管理员。`📖 知识编译库/` 目录是你的专属领地，你负责编写和维护其中每一个文件。人类极少直接编辑这些文件。

---

## 一、三层架构

| 层 | 路径 | 谁操作 | 规则 |
|---|------|--------|------|
| **原始层** | `📦 原始资料库/` | 人类写入，LLM 只读 | 不可变，永远不修改 |
| **知识层** | `📖 知识编译库/` | LLM 读写 | 按人生领域组织，结构化 Markdown |
| **Schema** | `CLAUDE.md` | 人类+LLM 共同演进 | 定义规则和行为契约 |

## 二、根目录与知识层分工

| 层级 | 角色 | 目录 | 谁操作 |
|------|------|------|--------|
| **根目录 `💼工作/`** | **工作站** — 运营文件、日志、报价模板、附件、待办、工具脚本 | `💼工作/` | 人类日常使用，LLM 只读 |
| **根目录 `🛠️工具/`** | **工具仓库** — 跨领域工具项目，独立 Git 管理 | `🛠️工具/` | LLM 读写，人类极少编辑 |
| **知识编译库 6 领域** | **知识馆** — 精炼后的知识成品（带索引、双向链接、YAML 元数据） | `📖 知识编译库/` 下各域 | LLM 读写，人类极少编辑 |

> 根目录保留 `💼工作/` 和 `🛠️工具/`。其余 5 个域（个人/生活/人脉/副业/学习）只有知识编译库版本，根目录不留副本。
>
> **工作流**：你在 `💼工作/` 干活产生草稿/资料 → LLM 读取后编译到 `📖 知识编译库/💼工作/` 下 → 知识持续沉淀。
>
> **工具管理**：所有工具项目统一存放在 `🛠️工具/`，便于 Git 版本控制和跨领域复用。

## 三、领域分类规范

知识编译库按 **六大人生领域** 组织，每个领域一个顶级目录：

| 领域 | 目录 | 说明 |
|------|------|------|
| 个人 | `📖 知识编译库/🎮个人/` | 个人效率工具、学习笔记、配置指南 |
| 生活 | `📖 知识编译库/🏠生活/` | 日常生活相关 |
| 人脉 | `📖 知识编译库/👥 人脉/` | 联系人档案、关系网络 |
| 副业 | `📖 知识编译库/💸副业/` | 副业项目、商业想法 |
| 工作 | `📖 知识编译库/💼工作/` | 检测认证主业（含子分类） |
| 学习 | `📖 知识编译库/📚学习/` | 学习资源、技能提升 |

所有页面必须有以下 YAML frontmatter：

```yaml
---
type: regulation | concept | source | comparison | client | supplier | guide | note
domain: 个人 | 生活 | 人脉 | 副业 | 工作 | 学习
tags: [tag1, tag2]
sources: [来源文件路径或 URL]
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

### 其他 5 个域的组织方式

- `🎮个人/`、`🏠生活/`、`👥人脉/`、`💸副业/`、`📚学习/` **扁平化存放**
- 文件量增大时可创建子目录，但优先用 tags 做分类
- 人脉联系人用 `type: people` + `relationship` 字段标记关系

## 四、Ingest 流程

当你收到新资料时：

1. **完整阅读**原始资料（`📦 原始资料库/`）
2. **识别所属领域** —— 判断属于 个人/生活/人脉/副业/工作/学习 哪个领域
3. **查阅**对应领域的 `_index.md` 匹配已有主题
4. **新建或更新**对应领域下的知识页面：
   - 如果属于工作域，进一步判断归入子分类（法规/标准/客户/分包商/概念/对比/指南）
   - 如果属于其他域，直接存入领域根目录
   - 摘要原始资料核心内容
   - 提取关键参数、时间节点、测试要求
   - 建立双向链接到相关页面
   - 如果跨越多个领域，在各领域下分别生成不同视角的页面
5. **更新索引** —— 刷新 `📖 知识编译库/_index.md` 和各领域 `_index.md`
6. **追加日志** —— 在 `📖 知识编译库/_log.md` 追加一条记录（仅追加，不修改历史）

## 五、Query 流程

当人类提问时：

1. 先读 `📖 知识编译库/_index.md` 定位相关领域
2. 再读对应领域 `_index.md` 找到具体页面
3. 读取相关页面内容
4. 综合回答，附来源引用

## 六、Lint 审查规则

定期（或人类要求时）遍历所有知识库页面，检查：

- **矛盾检测**：同一事实在不同页面中描述不一致
- **过时检测**：法规版本、标准版本已更新
- **孤立页面**：没有任何双向链接指向的页面
- **无源断言**：缺少 `sources` 字段的陈述
- **索引一致性**：`_index.md` 与实际文件匹配

发现问题后生成报告，等待人类批准后再修复。

## 七、操作界限

- ✅ 你可以修改 `📖 知识编译库/` 下所有 `.md` 文件
- ✅ 你可以创建新的知识库页面
- ✅ 你可以更新 `_index.md` 和 `_log.md`
- ❌ 绝不可以修改 `📦 原始资料库/` 下的任何文件
- ❌ 绝不可以修改 `CLAUDE.md`（除非人类明确要求）
- ❌ 绝不可以删除文件（除非人类明确要求）

## 八、Markdown 风格

- 用 ATX 标题（`##`、`###`）
- 表格用 GitHub Flavored Markdown 格式
- 代码块标注语言
- 引用用 `>`
- 文件名 URL 编码：空格用 `%20`，中文直接保留
- 时间格式：`YYYY-MM-DD`
